Supabase JWT payload shape (from Supabase GoTrue JWT):
The JWT issued by Supabase includes:
  payload.app_metadata.onboarded  ← boolean, written by admin.update_user_by_id
  payload.sub                      ← uid

In Next.js App Router middleware, Supabase SSR pattern:
```typescript
import { createServerClient } from "@supabase/ssr"

// In middleware:
const { data: { user } } = await supabase.auth.getUser()
// user?.app_metadata?.onboarded — check this after getUser()
// OR decode JWT directly via jose if getUser() is too heavy for edge middleware
```
Do NOT call `supabase.auth.getUser()` in middleware for each request (expensive). 
Prefer decoding the access token JWT claims directly via jose. Check:
  1. Is there a valid session cookie? No → redirect to TG bot URL
  2. Does decoded JWT have app_metadata.onboarded === true? No → redirect to /onboarding
  3. Yes → allow through

Port of the middleware change from Plan 01: Plan 01 only changed the unauthenticated redirect destination. 
Plan 03 ADDS the app_metadata.onboarded check for authenticated-but-not-onboarded users.

REQ-007 null-guard (GET /api/v1/user/me contract):
The backend exposes GET /api/v1/user/me returning user profile including telegram_id.
Response shape: { telegram_id: string | null, onboarding_status: string, ... }
The /onboarding page.tsx must on mount:
  1. Fetch GET /api/v1/user/me (using the session cookie — use fetch with credentials)
  2. If response is 401 OR parsed body has telegram_id === null → redirect to landing CTA ("/")
  3. If telegram_id is present → allow wizard to render
This prevents a user who somehow reached /onboarding without completing TG signup
from being stuck on a broken wizard state.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js middleware | JWT in session cookie crosses here; app_metadata claims checked |
| /onboarding page → GET /api/v1/user/me | Authenticated fetch; telegram_id null check guards wizard access |
| /auth/confirm → Supabase GoTrue | PKCE token_hash exchange; must remain autobind-free |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-DoS-1 | Denial of Service | middleware infinite redirect loop (/onboarding gate + unauthenticated gate) | mitigate | Middleware matcher MUST exclude /onboarding, /auth/*, and root / from the onboarded-check branch. Test 4 (non-dashboard-routes-not-gated) is the guard. |
| T-Tamper-2 | Tampering | app_metadata.onboarded JWT claim forgery from client | mitigate | app_metadata is only writable via admin.update_user_by_id (server-side); client JWTs signed by Supabase GoTrue; middleware verifies JWT signature before reading claims. |
| T-Info-2 | Information Disclosure | admit-list too permissive allows 'in_progress' users to reach AI pipeline | mitigate | Admit-list shrunk to ('completed',) only. test_message_handler_rejects_in_progress_status enforces. |
  | T-DoS-2 | Denial of Service | /onboarding null-guard redirect loop if /api/v1/user/me consistently returns null | mitigate | Redirect target is "/" (landing CTA), not /onboarding itself. No loop possible: landing CTA → TG bot flow → /onboarding (with telegram_id) → wizard. |
</threat_model>
