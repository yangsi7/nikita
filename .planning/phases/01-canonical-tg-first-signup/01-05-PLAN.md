Pattern 3 — Trigger with NULL guard (CRITICAL — prevents exception for non-TG signups):
```sql
-- Function: provisions public.users + child tables on email_confirmed_at flip
CREATE OR REPLACE FUNCTION auth.provision_user_on_confirm()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public, auth
AS $$
DECLARE
  v_telegram_id TEXT;
  v_user_id UUID;
BEGIN
  v_telegram_id := NEW.raw_user_meta_data->>'telegram_id';
  v_user_id := NEW.id;

  -- Insert user row (idempotent: DO UPDATE per FR-4b to update telegram_id on re-registration)
  INSERT INTO public.users (id, telegram_id, created_at, updated_at)
  VALUES (v_user_id, v_telegram_id::BIGINT, NOW(), NOW())
  ON CONFLICT (id) DO UPDATE SET
    telegram_id = EXCLUDED.telegram_id,
    updated_at = NOW();

  -- Insert user_profiles row (idempotent: DO NOTHING — do not overwrite existing onboarding state)
  INSERT INTO public.user_profiles (id, onboarding_status, created_at, updated_at)
  VALUES (v_user_id, 'pending', NOW(), NOW())
  ON CONFLICT (id) DO NOTHING;

  -- Insert user_metrics row (idempotent: DO NOTHING)
  INSERT INTO public.user_metrics (user_id, created_at, updated_at)
  VALUES (v_user_id, NOW(), NOW())
  ON CONFLICT (user_id) DO NOTHING;

  -- Insert user_vice_preferences row (idempotent: DO NOTHING)
  INSERT INTO public.user_vice_preferences (user_id, created_at, updated_at)
  VALUES (v_user_id, NOW(), NOW())
  ON CONFLICT (user_id) DO NOTHING;

  RETURN NEW;
END;
$$;

-- Trigger with NULL guard (fires ONLY when: was null, now not null, AND telegram_id present)
CREATE TRIGGER provision_user_on_email_confirm
  AFTER UPDATE OF email_confirmed_at ON auth.users
  FOR EACH ROW
  WHEN (
    OLD.email_confirmed_at IS NULL
    AND NEW.email_confirmed_at IS NOT NULL
    AND (NEW.raw_user_meta_data->>'telegram_id') IS NOT NULL
  )
  EXECUTE FUNCTION auth.provision_user_on_confirm();
```

FR-4b ON CONFLICT semantics (CRITICAL — do NOT use DO NOTHING for public.users):
- public.users: ON CONFLICT (id) DO UPDATE SET telegram_id = EXCLUDED.telegram_id, updated_at = NOW()
  Rationale: If a user re-registers with a new Telegram account, the telegram_id binding must update.
  DO NOTHING would silently retain the old telegram_id, breaking the association.
- Child tables (user_profiles, user_metrics, user_vice_preferences): ON CONFLICT DO NOTHING
  Rationale: These tables contain user state (onboarding progress, metrics, preferences) that must
  NOT be overwritten on re-trigger. Their data is cumulative, not a binding.

CRITICAL: The WHEN guard prevents the trigger from firing for non-TG signups (NULL telegram_id).
Without the WHEN guard, `v_telegram_id::BIGINT` would throw for non-TG users.

onboarding_status CHECK constraint (current → target):
Current (4 values): CHECK (onboarding_status IN ('pending', 'in_progress', 'completed', 'skipped'))
Target (3 values):  CHECK (onboarding_status IN ('pending', 'in_progress', 'completed'))
Migration: DROP CONSTRAINT, ADD CONSTRAINT.
Verify no existing rows have onboarding_status = 'skipped' before dropping:
  SELECT COUNT(*) FROM user_profiles WHERE onboarding_status = 'skipped';
If count > 0: UPDATE user_profiles SET onboarding_status = 'completed' WHERE onboarding_status = 'skipped'; first.
(Solo dev, no retained users — safe to wipe per ADR-220-8.)

telegram_signup_sessions and telegram_link_codes:
Check if these tables exist: mcp__supabase__list_tables
