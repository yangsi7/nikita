async def handle_email(self, chat_id: int, email: str) -> str:
    await supabase.auth.sign_in_with_otp({

Pattern 2 — handle_otp (verify + admin lock):
```python
async def handle_otp(self, chat_id: int, email: str, token: str) -> str:
    OTP_REGEX = re.compile(r"^[0-9]{8}$")  # 8 digits exactly per GH#431
    if not OTP_REGEX.match(token):
        return "Invalid code — paste the 8-digit code from your email"
    auth_resp = await supabase.auth.verify_otp({"email": email, "token": token, "type": "email"})
    uid = str(auth_resp.user.id)  # available after verify_otp succeeds
    # Admin lock — write to app_metadata (immutable from client per ADR-220-5)
    await supabase.auth.admin.update_user_by_id(uid, {"app_metadata": {"telegram_id": str(chat_id)}})
    # Generate portal link
    link_result = await supabase.auth.admin.generate_link({"type": "magiclink", "email": email})
    portal_url = f"{settings.portal_url}/auth/confirm?token_hash={link_result.properties.hashed_token}&type=email&next=/onboarding"
    return f"Registration complete! Open your portal: {portal_url}"
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram → signup_handler | User-supplied email and OTP code arrive via Telegram message text |
| signup_handler → Supabase admin API | app_metadata written via admin endpoint (server-side only) |
| wizard PATCH → Supabase admin API | onboarded flag written via admin endpoint outside transaction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-Spoof-1 | Spoofing | signup_handler: client could supply arbitrary telegram_id | mitigate | telegram_id written ONLY via admin.update_user_by_id to app_metadata (ADR-220-5). Client cannot write app_metadata via user-facing API. handle_email stashes in raw_user_meta_data (unverified); handle_otp overwrites with admin lock after OTP verification. |
| T-Tamper-1 | Tampering | Double OTP race: two verify_otp calls for same email | mitigate | SELECT FOR UPDATE in wizard PATCH serializes concurrent completions. Supabase GoTrue consumes OTP token on first verify_otp call — subsequent calls with same token fail. No code-level double-consume risk. |
| T-Tamper-2 | Tampering | Stale wizard PATCH (concurrent /onboarding PATCH calls) | mitigate | get_by_telegram_id_for_update uses SELECT FOR UPDATE row lock. Second concurrent request blocks until first commits. Idempotent: second call finds onboarding_status already 'completed' and returns 200 without re-writing. |
| T-EoP | Elevation of Privilege | user_metadata.role escalation attempt | mitigate | Arch-B NEVER writes to user_metadata for auth-sensitive fields. telegram_id and onboarded always go through admin.update_user_by_id to app_metadata. Code review gate: grep for user_metadata writes in signup_handler must return 0. |
| T-Info-1 | Information Disclosure | LLM-DEBUG log lines in telegram.py routes | mitigate | Plan 04 (PR-B) strips remaining PII log lines. Plan 02 removes lines 577-695 of telegram.py (FSM handler block) which contain the majority. |
</threat_model>
