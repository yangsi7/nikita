## Critical Finding F1 — Detail

### Repro evidence

POST `/api/v1/onboarding/answer` request body:
```json
{"slot_kind":"display_name","value":"WalkB3","turn_id":"7f45068b-ed38-40a3-9dc3-d0816f884d75","conversation_id":null}
```

BE response (200 OK):
```json
{
  "output": {
    "delta": {"kind": "display_name", "data": {"display_name": "WalkB3"}},
    "reply": "WalkB3, love that handle, feels like a username with a backstory. How old are you?",
    "next_slot_kind": "age",
    "kind": "success",
    "cohort_chips": null,
    "archetype_cards": null
  },
  "progress_pct": 7,
  "is_complete": false,
  "link_code": null,
  "conversation_id": "49fbac96-18d8-4ced-8f16-31089d101f71",
  "meta": {"source": "llm"}
}
```

Console error:
```
Uncaught Error: deriveAgentView: unhandled AnswerResponse kind: undefined
```

### Contract analysis

The FE `deriveAgentView` (per Spec 217-3B AC-13) expects a 6-kind discriminated union for the agent-view dispatcher. The BE response contains:

- `output.kind = "success"` (envelope status — gates success vs error, not agent-view)
- `output.delta.kind = "display_name"` (slot kind — what was extracted)
- Neither field is the 6-kind agent-view discriminator the FE wants.

Possibilities:
1. BE regression — never wired the agent-view kind field into the response payload
2. FE drift — reading the wrong field path (`response.kind` vs `response.output.kind` vs `response.output.agent_view.kind`)
3. Schema rename mid-development — BE renamed the field, FE not synced

Recommend the implementor:
- Read the FE `deriveAgentView` source to identify the exact field path being switched on
- Read the BE `nikita/api/routes/portal_onboarding.py` answer handler to identify the response builder
- Reconcile — pick one canonical name and emit/consume it consistently
- Add an integration test that asserts the BE response shape matches the FE Zod/Pydantic schema (this regression would have been caught pre-merge)

## Anomaly A1 — Magic-link → /dashboard not /onboarding

The Telegram inline portal-handoff button URL was:
```
https://nikita-mygirl.com/auth/confirm?token_hash=...&type=magiclink&next=/onboarding
```

After auth-confirm processed, the page landed on `/dashboard`, not `/onboarding` per the `next` query param. Two possibilities:
1. The `next` param is dropped by the auth-confirm route handler.
2. A higher-priority post-auth router rule routes verified-but-not-onboarded users to `/dashboard` (which then shows the "Welcome to Nikita's World — Start chatting on Telegram" empty state).

Either way: a user who clicks the email/Telegram CTA expecting to land in the wizard is dumped onto a dashboard that tells them to chat on Telegram. They have to know to manually navigate to `/onboarding` to access the wizard. This is a routing bug independent of 217-3B but discovered on the walk path.

Severity: MEDIUM (UX dead-end on the canonical signup path; not a crash). Manually walking around it during this audit was the only reason we reached the wizard at all.
